# KF Intelligent Customer Service Engine · HTTP API Reference

> Version: `0.2.0` · Protocol: HTTP/1.1 + JSON (UTF-8) · Streaming: SSE (`text/event-stream`)

---

## Overview

| Item | Description |
|------|-------------|
| Base URL | `http://<HOST>:9000` (production: via ALB Ingress, paths unchanged) |
| Encoding | Request and response are UTF-8. For POST JSON, explicitly set `Content-Type: application/json; charset=utf-8` |
| Streaming | SSE (`text/event-stream`), token-by-token delivery |
| Timestamps | All times in **UTC**, string format `YYYY-MM-DD HH:MM:SS` |
| Request Tracing | Every response includes `X-Request-ID` header; server logs correlate on this. 500 errors also return `request_id` in the body |
| Idempotency | `GET`/`DELETE` are idempotent; `POST /api/v1/workflows/{name}/run` creates a new turn on each call |
| Pagination | Uniform `limit` + `offset` query parameters |

---

## Authentication

Static API key allowlist, passed via request header:

```
X-API-Key: <YOUR_KEY>
```

- Key list configured in `config/auth.yaml` under `api_keys` (supports `${ENV_VAR}`)
- **Empty `api_keys` = auth disabled** (dev mode, all requests pass through)
- Skipped paths (`skip_paths`, no key required): `/health`, `/ready`, `/docs`, `/openapi.json`, `/redoc`
- Auth failure returns `401`:

```json
{ "error": "invalid or missing X-API-Key" }
```

> Interactive docs: the service includes Swagger UI at `GET /docs` and OpenAPI at `GET /openapi.json` (auto-generated by FastAPI), available for browser debugging.

## Error Handling

| Status | Meaning | Scenario |
|--------|---------|----------|
| `200` | Success | Normal response |
| `204` | Success, no content | Session deleted |
| `400` | Bad request | Session-workflow mismatch, invalid parameters, ingestion/search failure |
| `401` | Unauthorized | Missing or invalid `X-API-Key` |
| `404` | Not found | Workflow/session/collection/node not found, or session expired |
| `422` | Validation error | Missing required fields, empty `query`, type error |
| `500` | Internal server error | Unhandled exception (details only logged, not exposed; response includes `request_id`) |
| `503` | Config reload in progress | Hot reload window, retry after `Retry-After: 1` second |

**Business error** (HTTPException) format:

```json
{ "detail": "workflow 'foo' not found" }
```

**Unhandled exception** (500) format:

```json
{
  "error": "internal server error",
  "detail": "Server internal error, please try again later or contact administrator",
  "request_id": "3f8c1a2b..."
}
```

**Validation error** (422, FastAPI standard):

```json
{
  "detail": [
    {
      "type": "string_too_short",
      "loc": ["body", "query"],
      "msg": "String should have at least 1 character",
      "input": ""
    }
  ]
}
```

#### `POST /reload`

**Hot reload configuration**: re-reads all YAML config files and atomically replaces the running configuration without process restart.

- **Auth**: Yes
- **Request body**: empty JSON `{}` (POST requires a body)

**Response 200**

```json
{ "status": "ok", "workflows": ["auto_film", "customer_service", "default"] }
```

**How it works**:
1. Acquire write lock → mark `need_reload=True` on the old config
2. Scan `config/` and rebuild AppConfig (skipping `enabled: false` workflows)
3. Atomically replace global config → trigger DAG engine callback to update references
4. Release lock

**Conflict protection**: during the window from marking `need_reload` to completion (typically <100ms), all requests except `/reload` receive `503 Service Unavailable` + `Retry-After: 1`.

**Workflow enable/disable**: set `enabled: true|false` in `config/workflows/<name>/workflow.yaml` (default `true`). Set to `false`, then call `/reload` — the workflow disappears from the list and cannot be run. No file deletion required.

```powershell
Invoke-RestMethod -Uri "http://<HOST>:9000/reload" -Method Post -Headers @{"X-API-Key"="<KEY>"} -Body "{}"
```

#### `POST /api/v1/workflows/{name}/run` ⭐

**Main conversation engine entry point**. Executes one turn of the specified workflow (DAG: intent classification → routing → RAG retrieval → LLM generation).

- **Auth**: Yes
- **Path parameter**: `name` — workflow name
- **Query parameter**: `stream` (bool, default `false`) — when `true`, returns SSE stream

**Request Body** (`application/json`)

| Field | Type | Required | Description |
|-------|------|:--------:|-------------|
| `query` | string | ✅ | User input text (≥1 character) |
| `chat_id` | string | ✕ | Session ID for continuing a multi-turn conversation; omit to create a new session |
| `long_mem_data` | string | ✕ | Client-managed long-term memory, injected into the current turn's context |

**Response 200** (blocking mode, `stream=false`)

```json
{
  "chat_id": "chat_054d0450cd5a400dbb34",
  "turn_id": 1,
  "reply": "Haha, the price range for heat insulation film is quite wide, from a few hundred to several thousand or even tens of thousands..."
}
```

| Field | Description |
|-------|-------------|
| `chat_id` | Session ID, return this for multi-turn continuation |
| `turn_id` | Current cumulative turn count (starts at 1, increments) |
| `reply` | Final reply text |

**Response (streaming mode, `stream=true`)**

Returns `text/event-stream`, pushing tokens one by one.

Event sequence:
1. `status` (start) — `{"event":"status","node":"start","workflow":"auto_film"}`
2. Bare event line `event: node_start`
3. `token` (multiple) — `{"event":"token","data":"H"}`
4. Bare event line `event: node_end`
5. `done` (end) — `{"event":"done","chat_id":"…","turn_id":1,"reply":"full reply"}`
6. On error: `{"event":"error","data":"error message"}`

Raw SSE frame example:

```
data: {"event":"status","node":"start","workflow":"auto_film"}

event: node_start

data: {"event":"token","data":"H"}

data: {"event":"token","data":"e"}

event: node_end

data: {"event":"done","chat_id":"chat_x","turn_id":1,"reply":"Hello..."}
```

**Client parsing advice**: read JSON from `data:` lines, dispatch by `event` field; the `reply` in `done` is the authoritative full response (consistent with token-by-token concatenation).

**JavaScript Example**

```js
const resp = await fetch(`/api/v1/workflows/auto_film/run?stream=true`, {
  method: "POST",
  headers: { "Content-Type": "application/json", "X-API-Key": KEY },
  body: JSON.stringify({ query: "What does solar film do?", chat_id }),
});
const reader = resp.body.getReader();
const dec = new TextDecoder();
let buf = "";
while (true) {
  const { done, value } = await reader.read();
  if (done) break;
  buf += dec.decode(value, { stream: true });
  const lines = buf.split("\n"); buf = lines.pop();
  for (const line of lines) {
    if (!line.startsWith("data: ")) continue;
    const evt = JSON.parse(line.slice(6));
    if (evt.event === "token") process.stdout.write(evt.data);
    if (evt.event === "done") console.log("\n[done]", evt.chat_id, evt.turn_id);
    if (evt.event === "error") console.error("[error]", evt.data);
  }
}
```

**Errors**

| Code | Scenario |
|------|----------|
| `404` | Workflow not found; or session for `chat_id` not found/expired |
| `400` | `chat_id` belongs to a different workflow than the requested `name` |
| `422` | `query` is empty |

**Multi-turn conversation**: sessions are stored server-side (in-memory or Redis, with TTL). To continue, include the previous `chat_id` in the request body:

```bash
# Turn 1 (new session)
curl ... -d '{"query":"What does solar film do?"}'
# → {"chat_id":"chat_x","turn_id":1,"reply":"..."}

# Turn 2 (continuation, with context)
curl ... -d '{"query":"How about the price?","chat_id":"chat_x"}'
# → {"chat_id":"chat_x","turn_id":2,"reply":"..."}
```

Constraint: a `chat_id` is bound to the workflow it was created with; cross-workflow reuse returns `400`; if a session expires, returns `404`.

**Errors**

| Code | Scenario |
|------|----------|

### Document Ingestion

For building/updating the knowledge base. All requests support `rebuild=true` to drop and recreate the target collection first.

#### `POST /documents/upload`

Upload a single document for ingestion (auto chunking + vectorization).

- **Auth**: Yes
- **Content-Type**: `multipart/form-data`

**Form Fields**

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `file` | file | — (required) | Supports `.txt/.md/.pdf/.docx/.csv/.xlsx/.html` |
| `collection` | string | `default` | Target collection |
| `chunk_size` | int | 800 | Chunk size in characters |
| `chunk_overlap` | int | 64 | Overlap between chunks in characters |
| `rebuild` | bool | false | When `true`, drop the collection first then rebuild |

**Response 200**

```json
{
  "status": "ok",
  "file": "product_intro.md",
  "collection": "car_films",
  "chunks": 12,
  "rebuilt": false
}
```

```bash
curl -X POST "http://<HOST>:9000/documents/upload" -H "X-API-Key: <KEY>" \
  -F "file=@product_intro.md" -F "collection=car_films" -F "chunk_size=800"
```

#### `POST /export/chat.xlsx`

Export client-side chat history to Excel (`.xlsx`). Server generates a binary file stream.

- **Auth**: Yes

**Request Body** (`application/json`)

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `messages` | list[ChatMessage] | `[]` | Chat message list |
| `filename` | string | `chat_history` | Export filename (without extension) |

Each `ChatMessage`:

| Field | Type | Required | Description |
|-------|------|:--------:|-------------|
| `role` | string | ✕ | Role (e.g., `user`/`assistant`) |
| `content` | string | ✕ | Message content |
| `ts` | string | ✕ | Timestamp |
| `feedback` | string | ✕ | Rating (`up`/`down`) |
| `comment` | string | ✕ | Comment |
| `correction` | string | ✕ | Corrected answer |

- **Response**: `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`, `Content-Disposition: attachment`
- **Export columns**: `role, content, timestamp, feedback, comment, correction`

**Request Example**

```json
{
  "messages": [
    { "role": "user", "content": "Hello", "ts": "10:00:00" },
    { "role": "assistant", "content": "Hi, how can I help you?", "ts": "10:00:01",
      "feedback": "down", "comment": "Off-topic", "correction": "Should introduce the product" }
  ],
  "filename": "chat_history"
}
```

---

#### `POST /export/chat.csv`

Export client-side chat history to CSV (UTF-8 BOM, Excel-friendly). Request body is identical to `chat.xlsx`.

- **Auth**: Yes
- **Response**: `text/csv; charset=utf-8`, `Content-Disposition: attachment`
- **Export columns**: `role, content, timestamp, feedback, comment, correction`
